⚕ ER Triage AI Co-pilot — System Architecture

Current build state — Phase 5 (Enterprise AI Systems), no EMR integration yet 🖥 Frontend: Vanilla HTML/CSS/JS ⚙ Backend: N8N Cloud 🤖 AI: OpenAI GPT-4o 🔎 RAG: Pinecone 🗄 DB: Supabase
⚠ Updated 2026-07-10 to match the live build. Original design (fan-out WF-1/WF-2 inside one N8N workflow, EMR read/write, queue polling) was superseded — N8N does not execute fanned-out branches concurrently, which is why the architecture split into independent webhooks parallelized from the frontend, then evolved further into the single WF5 agentic orchestrator described below.

⚙ N8N
Workflow
Engine
Parse Complaint Extraction
  • Webhook trigger (/parse-complaint)
  • Length-guard IF node
  • OpenAI: extraction prompt, temp 0.2, JSON mode
  • Return: {structured: {category, onset, duration, symptoms[], trigger, context}}
WF5 Orchestrator (Agentic)
  • Webhook trigger (/orchestrate-triage)
  • Layer-2 guardrail: GPT-4o content classifier → blocks non-clinical/injection input, logs to guardrail_blocks
  • Quick Classify: ESI tier (Critical/Urgent/Low) + primary_flag — hard overrides for STEMI, FAST-positive stroke → ESI 1
  • RAG: embed complaint_text + primary_flag + complaint_summary (never raw vitals) → Pinecone topK=4 (Critical) / topK=2 (Urgent)
  • Build Clinical Brief: GPT-4o + retrieved guidelines → esi_level, disposition, time_targets, structured actions (source-cited, drug/dosage banned)
  • Merge Usage: sums tokens across every OpenAI call in the request
  • Supabase upsert: encounters (no PII) + latency/token columns
Ingest Knowledge Base
  • wf-ingest-and-retrieve.json — run on-demand, not per-request
  • Embeds 24-doc clinical knowledge base
  • Writes embeddings to Pinecone index
  • Re-run required whenever KB docs change
3
WF5 makes 2–3 OpenAI calls per request (guardrail + classify + brief) plus a Pinecone embedding call — token usage across all of them is summed and attached to the response
🤖 OpenAI
Agent
GPT-4o
Structured Extraction GPT-4o · temp 0.2 · JSON mode
Used by
/parse-complaint (independent of WF5)
Output (JSON)
{ "structured": { "complaint_category", "onset", "duration",
"symptoms": [], "trigger", "context" } }
Layer-2 Guardrail Classifier GPT-4o · temp 0 · JSON mode
Purpose
Content-safety check — non-clinical input, prompt injection, out-of-scope requests. 16/16 test cases passing.
Output (JSON)
{ "blocked": true/false, "message": "..." }
Quick Classify GPT-4o · JSON mode
Purpose
Routes to Critical / Urgent / Low branch. ESI criteria embedded in system prompt. Hard "always ESI 1" overrides for STEMI and FAST-positive stroke — do-not-downgrade instruction.
Output (JSON)
{ "esi_level": 1, "primary_flag": "stroke" }
Build Clinical Brief GPT-4o · JSON mode · RAG-grounded
Purpose
Combines classify result + Pinecone-retrieved guidelines into the final response. Drug/dosage recommendations banned; every action carries a source citation.
Output (JSON)
{ "esi_level", "acuity", "disposition", "confidence",
"primary_flag", "time_targets", "clinical_protocol",
"immediate_actions": [{action, category, source}],
"retrieved_guidelines": [], "usage": {prompt_tokens, completion_tokens} }
ESI Routing
ESI ≤ 2 → critical (red, submit-gated). ESI 3 → urgent (amber). ESI 4–5 → low (no flag).
Clinical Action Safety
Drug/dosage recommendations banned. Actions are structured {action, category, source} — source citation required.
PII Scope
Name/MRN collected in the form for display + prompt context only — never written to Supabase (Option B, R4-4).
Nurse Override
Accept/dismiss logged to nurse_actions. Field-level edit capture (AI value vs. nurse-changed value) not yet built (P5-2, partial).
4
No EMR integration exists in this build. WF5 writes directly to Supabase — encounters (de-identified) and nurse_actions (audit). External EHR/FHIR integration (P6-2) is on hold, not started.
